# Terminal View

The Terminal tab is the power-user escape hatch. Full access to raw iptables state, the packet flow tracer, and export tools.

Can be opened as a **split panel** alongside Rules.

## Layout

```
┌─────────────────────────────────────────────────────────┐
│  web-01                                    ● Connected  │
│                                                         │
│  [ Rules ]  [ Activity ]  [ Terminal ]                  │
│  ─────────────────────────────────────────────────────  │
│                                                         │
│  [ Raw Rules ]  [ Packet Tracer ]  [ SSH Log ]          │
│                                                         │
└─────────────────────────────────────────────────────────┘
```

## Raw Rules Sub-Tab

Shows the full `iptables-save` output in an editable code editor (CodeMirror 6).

```
┌─────────────────────────────────────────────────────────┐
│  Raw Rules                     [Copy] [Export ▾] [Reset] │
│  ─────────────────────────────────────────────────────  │
│  # Generated by iptables-save v1.8.7                    │
│  *filter                                                │
│  :INPUT DROP [412:24720]                                │
│  :FORWARD DROP [0:0]                                    │
│  :OUTPUT ACCEPT [1893:284700]                           │
│  -A INPUT -i lo -j ACCEPT                               │
│  -A INPUT -m conntrack --ctstate RELATED,ESTABLISHED    │
│     -j ACCEPT                                           │
│  -A INPUT -p tcp -m multiport --dports 80,443           │
│     -j ACCEPT                                           │
│  -A INPUT -p tcp --dport 22 -s 83.12.44.0/24           │
│     -j ACCEPT                                           │
│  -A INPUT -p tcp --dport 9100 -s 10.0.0.0/24           │
│     -j ACCEPT                                           │
│  -A INPUT -p icmp -j ACCEPT                             │
│  COMMIT                                                 │
│  ─────────────────────────────────────────────────────  │
│                                                         │
│  ⚠️ Editing here modifies iptables rules directly.      │
│  Changes still go through the safety timer.             │
│                                                         │
│                         [Apply Raw Changes]             │
└─────────────────────────────────────────────────────────┘
```

### Features

- **Syntax highlighting** for iptables-save format
- **Line numbers**
- **Editable** — power users can paste rulesets directly
- Changes go through the same safety flow (auto-revert timer)
- **System chains** (Docker, fail2ban, K8s) are highlighted with a background tint and marked "(system — read-only)"

### Export Menu

The `[Export ▾]` button provides:

```
┌─────────────────────────────────────┐
│  Copy as iptables-save format       │
│  Copy as shell script               │
│  Copy as Ansible playbook           │
│  Download as file                   │
└─────────────────────────────────────┘
```

**Ansible playbook export** generates a ready-to-use playbook:
```yaml
- name: Configure firewall on web-01
  hosts: web-01
  tasks:
    - name: Apply iptables rules
      community.general.iptables_state:
        state: restored
        path: /etc/iptables/rules.v4
```

**Shell script export** generates `iptables` commands:
```bash
#!/bin/bash
iptables -F INPUT
iptables -P INPUT DROP
iptables -A INPUT -i lo -j ACCEPT
iptables -A INPUT -m conntrack --ctstate RELATED,ESTABLISHED -j ACCEPT
# ...
```

### Table Filter

```
  Show: [All ▾]  |  filter  |  nat  |  mangle  |  raw
```

## Packet Tracer Sub-Tab

The visual flow tracer — a **read-only diagnostic tool** (not an editor).

### Input Form

Horizontal layout — all fields in one row for compactness. Labels below fields (Apple Maps pattern — value is primary, label is secondary).

```
│  Packet Tracer                                                          │
│                                                                         │
│  Simulate a packet to see how your rules handle it.                     │
│                                                                         │
│  ┌──────────────┐ ┌──────────────┐ ┌──────┐ ┌──────┐ ┌──────────┐     │
│  │ 45.33.12.8   │ │ 10.0.1.5     │ │ TCP  │ │  22  │ │ Incoming │     │
│  │ Source IP     │ │ Dest IP      │ │Proto │ │ Port │ │Direction │     │
│  └──────────────┘ └──────────────┘ └──────┘ └──────┘ └──────────┘     │
│                                                                         │
│  State: [New ▾]                                                         │
│                                                                         │
│  Quick:  SSH from my IP    Web traffic    Ping              [Trace ▶]  │
```

- Quick test chips pre-fill all fields and auto-run the trace
- Connection state dropdown: New / Established / Related / Invalid

### Packet Paths

The tracer correctly models **all three** netfilter paths:

1. **Incoming to local**: raw PREROUTING → conntrack → mangle PREROUTING → nat PREROUTING → **routing decision (local)** → mangle INPUT → filter INPUT
2. **Forwarded**: raw PREROUTING → conntrack → mangle PREROUTING → nat PREROUTING → **routing decision (forward)** → mangle FORWARD → filter FORWARD → mangle POSTROUTING → nat POSTROUTING
3. **Locally generated**: raw OUTPUT → conntrack → mangle OUTPUT → nat OUTPUT → **routing decision** → filter OUTPUT → mangle POSTROUTING → nat POSTROUTING

The routing decision diamond is a critical node — it determines whether the packet goes to INPUT or FORWARD. This is shown in the expanded full-path view.

### Connection State Toggle

```
  Connection state: [New ▾]
                     ├ New (SYN)
                     ├ Established (ACK)
                     ├ Related
                     └ Invalid
```

A SYN packet hits different rules than an ESTABLISHED ACK. The tracer must model conntrack state to give accurate results.

### Visualization Style

- **Chain nodes**: Rounded rectangles, 160px wide (room for rule listings), #F5F5F7 fill, 8px radius
- **Matching rule**: 10% status color background + 1.5px left border + pulse dot (600ms)
- **Near-miss annotation**: "src ≠" in 10px #FF9500 showing why a close match failed
- **Blocked path**: Red X at blocking node, subsequent edges as 1px dashed #C7C7CC
- **Empty chains**: "(empty)" in 11px italic #AEAEB2, reduced height (48px)
- **Edges**: 2px solid with 6px arrowheads. Green (#34C759) for traversed, dashed grey after block
- **Routing decision**: Diamond shape node between PREROUTING and INPUT/FORWARD

### Custom Chain Jumps

When the packet jumps to a custom chain:

```
  INPUT ──▶ MY_WEB_RULES ──▶ (RETURN) ──▶ back to INPUT
```

The flow branches to show the jump visually.

### Packet Traversal Animation

1. After "Trace", the decision chain builds over 1.2s. Each rule line fades in sequentially (100ms per rule)
2. A bright dot (8px, status color, glow filter) travels through the chain, pausing at each rule (100ms)
3. At the matching rule, the dot stops and pulses once (600ms)
4. "Why" explanation fades in last (200ms, 300ms after dot stops)
5. Total: ~2-3 seconds. "Skip animation" link available
6. After 5+ traces in session, animation reduces to simple highlight (power users don't need the show)
7. In full-path mode, the dot travels across all chain nodes sequentially

### Corner Cases

**No rules at all:**
```
  Result: ✅ ACCEPTED (default policy)
  This host has no firewall rules. All traffic accepted.
  [Set up firewall rules]
```

**FORWARD packet on non-routing host:**
```
  ⚠️ IP forwarding is not enabled.
  Packets would be dropped by the kernel before reaching iptables.
  Showing what would happen IF forwarding were enabled.
```
